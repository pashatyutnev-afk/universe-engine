# UNIVERSE ENGINE — ARCHITECTURE OVERVIEW
## Обзор архитектуры системы
## Уровень: L1 · Reference

---

## 0. НАЗНАЧЕНИЕ ДОКУМЕНТА

Этот документ даёт целостное понимание архитектуры Universe Engine:
- из каких логических частей состоит система
- как они связаны между собой
- как данные проходят через систему
- где находится логика, а где — данные

Документ предназначен для:
- быстрого погружения нового чата
- навигации по системе
- проверки архитектурной целостности

---

## 1. ОБЩАЯ ИДЕЯ АРХИТЕКТУРЫ

Universe Engine построен как система с внешней памятью и жёстким разделением ролей.

Ключевая идея:
> Память вынесена наружу,  
> логика формализована,  
> контекст не хранится в чатах.

Система работает не «в диалоге», а **по карте**.

---

## 2. ОСНОВНЫЕ ЛОГИЧЕСКИЕ БЛОКИ

Архитектура состоит из следующих логических слоёв:

1. Standards & Canon  
2. Index Layer  
3. System Entities  
4. Models  
5. Knowledge Base  
6. Projects  
7. Output  

Каждый слой выполняет **строго свою функцию**.

---

## 3. STANDARDS & CANON

**Назначение:** законы системы.

- определяют правила
- фиксируют терминологию
- задают границы допустимого
- не зависят от проектов

Без этого слоя система теряет устойчивость.

---

## 4. INDEX LAYER

**Назначение:** навигация и сценарий.

INDEX:
- не хранит данные
- не дублирует файлы
- описывает связи и процессы

INDEX — это:
- карта системы
- интерфейс между человеком и ИИ
- единый источник логической истины

---

## 5. SYSTEM ENTITIES

**Назначение:** активные участники системы.

Сущности:
- принимают решения
- фильтруют данные
- применяют модели
- валидируют результат

Категории сущностей:
- Engines
- Orchestrators
- Specialists
- Controllers
- Validators
- Quality

Сущности не хранят знания — они **работают с ними**.

---

## 6. MODELS

**Назначение:** способы применения знаний.

Модели:
- описывают «как делать»
- не принимают решений
- выбираются в процессе работы

Модель — это мост между знаниями и действиями.

---

## 7. KNOWLEDGE BASE

**Назначение:** пассивная энциклопедия.

Knowledge Base:
- хранит мировые стандарты
- содержит best practices
- не влияет на процесс напрямую

Знания используются только через модели и сущности.

---

## 8. PROJECTS

**Назначение:** вход системы.

Проекты:
- содержат цели и требования
- задают ограничения
- являются единственным источником входных данных

Система ничего не добавляет от себя.

---

## 9. OUTPUT

**Назначение:** результат работы.

Output:
- формируется системой
- не влияет обратно без фиксации
- используется как финальный продукт

---

## 10. ПОТОК ДАННЫХ (УПРОЩЁННО)

Projects  
↓  
Index  
↓  
System Entities  
↓  
Models  
↓  
Knowledge Base  
↓  
System Entities  
↓  
Output


Этот поток — базовый для всех процессов.

---

## 11. ПОЧЕМУ АРХИТЕКТУРА УСТОЙЧИВА

Система устойчива, потому что:
- память вынесена в файлы
- логика формализована
- роли разделены
- изменения фиксируются
- чаты не являются хранилищем

Это позволяет системе расти без потери смысла.
